
## 静态 vs 动态的本质区别

Hexo 生成的确实全是静态 HTML 文件。理解这个概念很重要：

**静态网站：**

```
用户访问页面
    ↓
服务器直接返回一个现成的 HTML 文件
    ↓
完成，没有任何计算
```

**动态网站（比如微博、知乎）：**[[dynamic blog动态blog]]


```
用户访问页面
    ↓
服务器运行程序，查询数据库，实时拼装 HTML
    ↓
返回给用户
```

Hexo 的工作是在你的电脑上提前把所有页面算好，生成一堆 `.html` 文件，上传到 GitHub，GitHub 只负责原样发给访问者。服务器不需要运行任何程序，这也是为什么 [[GitHub Pages]] 可以免费托管的原因——它只是在提供文件下载服务。

## hexo s 的作用---本地开放端口预览

`hexo s`（server）就是纯预览用的 ==修改可以实时刷新查看==
它在你本地启动一个临时服务器，让你能在浏览器里看效果，不会影响线上任何东西。


|命令|作用|影响线上吗|
|---|---|---|
|`hexo s`|本地预览，边改边看|不影响|
|`hexo g`|生成静态文件到 public/|不影响|
|`hexo d`|把 public/ 推送到 GitHub|这步才影响线上|

`hexo s` 有一个比 `hexo g` 更方便的地方：它会监听文件变化，你改了 Markdown 文件保存，刷新浏览器就能看到更新，不需要手动重新生成。所以写作时的工作流是：

```
开一个终端跑 hexo s，浏览器开着本地预览
    ↓
Obsidian 里写文章，随时保存
    ↓
浏览器刷新看效果
    ↓
满意了再跑 hexo g && hexo d 发布到线上
```

## 静态网站的限制

既然全是静态文件，有些功能天然做不了：

**做不了的：**

- 用户注册登录
- 评论系统（需要存储数据）
- 实时更新内容

**绕过方案：**

- 评论系统用第三方服务（Waline、Giscus），把数据存在别人的服务器上，你的页面只是嵌入一个 iframe
- 搜索功能用本地搜索插件，把索引在 `hexo g` 时预先生成成一个 JSON 文件，搜索在浏览器里完成

这也是为什么说 Hexo 适合博客但不适合做复杂应用的原因——它的边界就是"展示内容"这件事。
