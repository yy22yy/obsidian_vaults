## 一、[[DNS]]*缓存*中毒（DNS *Cache* Poisoning）

### 1. DNS系统的基本结构

在互联网中，域名解析流程大致为：

```
用户程序 → 本地DNS缓存 → 本地DNS服务器 → 递归/权威DNS服务器 → 返回IP
```

DNS服务器为了减少查询延迟，会 **缓存解析结果**：

```
example.com → 93.184.216.34
TTL = 3600s
```

在 TTL 时间内再次访问时，不再查询权威服务器，而是直接使用缓存。

---

### 2. DNS缓存中毒的核心思想

==**攻击目标：让DNS服务器缓存错误的IP映射。**==

正常缓存：

```
bank.com → 真实IP
```

被污染缓存：

```
bank.com → 攻击者IP
```

结果：

```
用户访问 bank.com
↓
DNS返回攻击者IP
↓
用户连接到假网站
```

这就是 **DNS缓存中毒**。

---

### 3. 攻击原理（技术本质）

DNS使用 **UDP协议（无连接）**。

查询过程：

```
DNS Query
客户端 → DNS服务器
Transaction ID = 1234
```

DNS服务器等待返回：

```
DNS Response
Transaction ID = 1234
```

攻击者可以：

1. 监听或猜测 **Transaction ID**
    
2. 伪造 **DNS响应包**
    
3. 抢在真正服务器之前发送
    

如果成功：

```
伪造DNS响应
example.com → 攻击者IP
TTL = 1 day
```

DNS服务器会：

```
信任这个响应
并缓存它
```

于是污染发生。

---

### 4. 为什么早期DNS容易被攻击

因为DNS设计时没有安全机制：

|问题|结果|
|---|---|
|UDP无连接|无法确认响应来源|
|Transaction ID只有16bit|容易猜测|
|没有加密|可伪造|
|DNS缓存机制|错误结果会持续|

现代数据包加密？DNS的数据包怎么样的？DNS的缓存机制具体是怎么样的？

---

### 5. 典型攻击流程

```
攻击者 → 向DNS服务器触发查询
            ↓
       DNS服务器发起查询
            ↓
攻击者大量伪造响应
TransactionID=随机猜测
            ↓
若猜中
DNS服务器缓存错误IP
            ↓
用户访问域名
被引导到恶意服务器
```

---

### 6. 防御方法

常见防御：

| 方法      | 原理      |
| ------- | ------- |
| 随机端口    | 增加猜测难度  |
| DNSSEC  | DNS数字签名 |
| 限制递归DNS | 防止外部利用  |
| 缩短TTL   | 减少污染时间  |

DNSSEC通过 **公钥签名** 确保：

```
DNS记录 = 可验证真实性
```
