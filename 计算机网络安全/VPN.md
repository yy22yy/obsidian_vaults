


## [[IPsec]] 在 VPN 中的作用

IPSec（Internet Protocol Security）是网络层安全协议族，为 IP 数据包提供**机密性、完整性、抗重放、数据源认证**四项核心安全保障，是构建 VPN 的主流技术基础。

VPN 的本质需求是：在不可信的公网上，建立逻辑上隔离、安全的私有通信隧道。IPSec 恰好在网络层解决这个问题，对上层协议透明。

---

## IPSec 的协议组成

IPSec 不是单一协议，而是协议族：

|组件|功能|
|---|---|
|**AH**（Authentication Header）|完整性 + 数据源认证，不加密|
|**ESP**（Encapsulating Security Payload）|加密 + 完整性 + 数据源认证（实践中主用）|
|**IKE**（Internet Key Exchange）|自动协商 SA、交换密钥（基于 DH）|
|**SA**（Security Association）|单向的安全上下文，描述双方约定的算法、密钥、生命周期|

---

## 两种工作模式

### 传输模式（Transport Mode）

- 仅加密/认证 IP 载荷（TCP/UDP payload），保留原始 IP 头
- 适用：**端到端**场景，两端设备本身就是通信主体
- 典型场景：主机 A ↔ 主机 B 直接建立 IPSec

```
原始包：[IP头: A→B] [TCP] [Data]
传输模式：[IP头: A→B] [ESP头] [加密(TCP+Data)] [ESP尾+Auth]
```

### 隧道模式（Tunnel Mode）

- 将整个原始 IP 包封装，外层新增新 IP 头
- 适用：**网关到网关**或**客户端到网关**场景
- 原始包对外完全不可见，隐藏了内网拓扑

```
原始包：[IP头: 192.168.1.x→10.0.0.x] [TCP] [Data]
隧道模式：[新IP头: GW1→GW2] [ESP头] [加密(原始IP头+TCP+Data)] [ESP尾+Auth]
```

## 不同场景下 IPSec 如何实现 VPN

### 场景一：Site-to-Site VPN（站点间）

```
分支机构                         总部
[PC] → [路由器/防火墙 GW1] ——公网—— [GW2 路由器/防火墙] → [服务器]
            ←——— IPSec 隧道（隧道模式）———→
```

**实现过程：**

1. GW1 与 GW2 通过 IKE Phase 1 建立 ISAKMP SA（认证身份，协商加密参数），常用预共享密钥或证书
2. IKE Phase 2 协商 IPSec SA（ESP 参数、感兴趣流 ACL）
3. 分支 PC 发出的包到达 GW1，匹配感兴趣流 → 触发 ESP 封装（隧道模式）→ 送往 GW2
4. GW2 解封装、验证 → 转发至内网服务器
5. 内网 PC 无需任何 IPSec 配置，对其完全透明

**关键点：** SA 是单向的，双向通信需要两对 SA。

---

### 场景二：Remote Access VPN（远程接入）

```
出差员工                              企业
[笔记本 IPSec 客户端] ——公网—— [VPN 网关] → [内网]
```

**实现方式：**

- 客户端运行 IPSec 软件（如 Cisco AnyConnect、strongSwan）
- 使用隧道模式，客户端获得一个虚拟内网 IP（通过 IKEv2 Config Payload 或 XAUTH 分配）
- 所有流量先加密送往网关，网关解封装后进入内网

**与 Site-to-Site 的差异：** 客户端 IP 动态变化，IKE 需支持 NAT-T（NAT 穿越，将 ESP 封装进 UDP 4500 端口）以应对客户端在 NAT 后面的情况。

---

### 场景三：Host-to-Host（端到端，较少用于 VPN）

使用传输模式，两台主机直接建立 IPSec。常见于服务器间的加密通信（如数据库服务器与应用服务器），不需要隧道，开销更小。

---

## IPSec 与网关、防火墙的关系

这是理解部署架构的关键，三者关系取决于**IPSec 终结点在哪里**。

### 关系一：防火墙自身作为 IPSec 网关（最常见）

```
[公网] → [防火墙（同时运行 IPSec）] → [内网]
```

防火墙同时承担两个角色：

- **IPSec 终结点**：负责加解密、SA 管理
- **访问控制**：对解密后的明文流量做策略过滤

**处理顺序（入方向）：** 收到加密包 → ESP 解封装 → 明文包 → 防火墙规则检查 → 转发内网

**处理顺序（出方向）：** 内网包 → 防火墙规则检查 → 匹配感兴趣流 → ESP 封装 → 送出公网

这是企业边界防火墙（如 Cisco ASA、FortiGate、华为 USG）的典型部署方式。

---

### 关系二：IPSec 网关在防火墙内侧（DMZ 部署）

```
[公网] → [外层防火墙] → [DMZ: VPN 网关] → [内层防火墙] → [内网]
```

外层防火墙只放行 IKE（UDP 500/4500）和 ESP（协议号 50）流量，其余拒绝。VPN 网关在 DMZ 中终结 IPSec，解密后的流量再经内层防火墙过滤进入内网。

**优点：** VPN 网关即使被攻破，内层防火墙仍保护内网；职责分离更清晰。

---

### 关系三：防火墙对 IPSec 流量的透传（Pass-through）

当 IPSec 网关在防火墙内侧时，防火墙不参与 IPSec 处理，仅做透传：

- 放行 **UDP 500**（IKE 协商）
- 放行 **UDP 4500**（NAT-T）
- 放行 **ESP（IP 协议号 50）** 和 **AH（IP 协议号 51）**

此时防火墙看不到加密载荷内容，无法对内层流量做深度检测（DPI），这是一个安全权衡。

---

### 核心矛盾：加密 vs 检测

|情况|防火墙能否检测内层流量|
|---|---|
|防火墙 = IPSec 终结点|✅ 解密后可检测|
|IPSec 终结点在防火墙后|❌ 防火墙只见密文|
|SSL VPN（替代方案）|✅ 可在应用层终结，便于检测|

这也是部分企业转向 SSL VPN 或 ZTNA 的原因之一——IPSec 的网络层加密对防火墙的深度检测不友好。

---

## 总结

IPSec 通过 **ESP 隧道模式** 解决 VPN 的核心需求（加密隧道 + 身份认证），通过 **IKE** 解决密钥管理问题。与防火墙的关系本质上是**安全功能边界的划分问题**：IPSec 负责传输层面的机密性与完整性，防火墙负责访问控制与流量过滤，两者在现代网络设备中往往集成在同一硬件上，但逻辑上职责清晰分离。
