## TCP利用缓存保证传输稳定性
TCP采用这样的方式确认对方是否收到了数据，在得到对方确认之前，发送过的包都会保存在发送缓冲区中。如果对方没有返回某些包对应的ACK号，那么就重新发送这些包。

“序号”和“ACK号”可以确认接收方是否收到了网络包

# 为什么TCP比UDP更容易受到攻击？
**总结对比表：**

| 攻击维度 | TCP       | UDP       |
| ---- | --------- | --------- |
| IP伪造 | 三次握手阻断    | 直接可行      |
| 放大攻击 | 不可行       | 核心载体      |
| 洪泛效率 | 受拥塞控制限制   | 全速发包      |
| 状态过滤 | 可基于连接状态过滤 | 无状态，难过滤   |
| 协议滥用 | 握手验证拦截大部分 | 任意数据报均被处理 |

### 请求/响应不对称攻击---小带宽放大攻击

```
攻击者（真实IP: A）
    ↓ 发送小请求，源IP伪造为受害者IP: V
公共UDP服务器（DNS/NTP等）
    ↓ 向V发送大响应（因为它以为请求来自V）
受害者（IP: V）← 被海量大包淹没
```

所以不是"两次攻击"，而是：

1. **攻击者只发一个小包**，自己几乎不消耗带宽
2. **UDP服务器成为攻击的武器**（被称为reflector/反射器），替攻击者向受害者发送大包
3. **受害者收到的是合法服务器的响应**，很难在IP层过滤（因为源IP是真实的DNS/NTP服务器）

---

**"两个攻击目标"的理解：**

你说的"两次攻击无辜者"也有一定道理——实际上有**两个受害方**：

- **受害者V**：被海量响应包打垮（DDoS目标）
- **UDP服务器**：被攻击者滥用，消耗其出口带宽和处理资源，它也是无辜受害方

---

**放大的关键在于比例**，不对称性体现在：

- 攻击者发 1 字节的查询
- 服务器返回 100 字节的响应给受害者

==攻击者用极小的带宽，撬动公网上大量服务器的带宽，集中打击一个目标。这就是"放大"的本质。==
