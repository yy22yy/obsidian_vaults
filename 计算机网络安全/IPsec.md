## IPsec 介绍

[[SA]]：[[IKE]]协议用于动态建立SA

[[VPN]]

两个主要封装协议：[[AH]]，[[ESP]]
![[Pasted image 20260311161220.png]]


IPsec是工作在**网络层（L3）**的安全协议族，为IP数据包提供认证、完整性、加密保护。与TLS不同，IPsec对上层应用透明，不需要应用程序做任何修改。

---

### 两个核心子协议

**AH（Authentication Header）**

- 提供：数据完整性 + 源认证 + 抗重放
- 不提供：加密（明文传输）
- 覆盖范围：IP头 + 载荷（因此NAT会破坏AH，因为NAT修改IP头会导致校验失败）

**ESP（Encapsulating Security Payload）**

- 提供：加密 + 完整性 + 源认证 + 抗重放
- 覆盖范围：仅载荷（不覆盖外层IP头，NAT兼容）
- 实际部署中ESP使用更广泛

---

### 两种工作模式

**传输模式（Transport Mode）**

```
[原IP头] | [AH/ESP头] | [原载荷]
```

保护IP载荷，原IP头保留，端到端通信，适合主机到主机。

**隧道模式（Tunnel Mode）**

```
[新IP头] | [AH/ESP头] | [原IP头] | [原载荷]
```

整个原始IP包被封装，加上新IP头，原IP头也受保护，适合VPN网关到网关。

## 模式和协议的复合

```text
传输模式AH：  [原IP头] [AH] [载荷]
              ←认证覆盖整个包→

传输模式ESP： [原IP头] [ESP头] [加密载荷] [ESP尾/认证]
                       ←加密→←认证→

隧道模式AH：  [新IP头] [AH] [原IP头] [载荷]
              ←认证覆盖整个包→

隧道模式ESP： [新IP头] [ESP头] [加密(原IP头+载荷)] [ESP尾/认证]
                       ←加密→←认证→
```

# ==两个模式和网关问题==

### 问题的直觉根源

网关的本职工作是**转发**，不是通信端点。它收到包之后必须拆开IP头，看目的地址，再决定往哪转发。

传输模式下，载荷被加密了，但**原IP头是明文暴露的**：

```
[原IP头: HostA → HostB私有IP] [ESP头] [加密载荷]
```

网关A把这个包发出去，公网路由器看到目的地址是`192.168.2.20`（私有IP），**根本不认识这个地址，包直接被丢弃**。

包还没到网关B，就已经死在公网上了。

---

### 隧道模式解决的正是这个问题

把整个原始包（含原IP头）加密，套上新IP头：

```
[新IP头: 网关A公网IP → 网关B公网IP] [ESP头] [加密(原IP头HostA→HostB + 载荷)]
```

公网路由只看新IP头，按网关A→网关B正常转发。到了网关B，拆掉新IP头，解密，还原出原始包，再按原IP头转发给HostB。

---

### 一张图说清楚

```
传输模式（网关场景，失败）：

HostA → 网关A → [192.168.2.20][ESP][加密] → 公网路由器 → ❌丢弃
                  ↑
              私有IP，公网不可路由


隧道模式（网关场景，成功）：

HostA → 网关A → [网关B公网IP][ESP][加密(原包)] → 公网路由器 → 网关B → 解封装 → HostB
                  ↑
              公网IP，正常路由
```

---

所以本质就是你说的：**网关需要一个可路由的公网IP头来完成转发，传输模式没有提供这个，隧道模式通过套一层新IP头解决了这个问题。**
