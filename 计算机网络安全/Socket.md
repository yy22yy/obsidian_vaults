# 本质理解
套接字的实体就是通信控制信息

本来套接字就只是一个概念而已，并不存在实体，如果一定要赋予它一个实体，我们可以说这些控制信息就是套接字的实体，或者说存放控制信息的内存空间就是套接字的实体。

套接字的内容---基本的ip、port；等待信息，接受信息

协议栈是根据套接字中记录的控制信息来工作的。

图中每一行相当于一个套接字，当创建套接字时，就会在这里增加一行新的控制信息，赋予“即将开始通信”的状态，并进行通信的准备工作，如分配用于临时存放收发数据的缓冲区空间。

![[Pasted image 20260305131556.png]]

>协议栈首先会分配用于存放一个套接字所需的内存空间。用于记录套接字控制信息的内存空间并不是一开始就存在的，因此我们先要开辟出这样一块空间来[插图]，这相当于为控制信息准备一个容器。但光一个容器并没有什么用，还需要往里面存入控制信息。套接字刚刚创建时，数据收发操作还没有开始，因此需要在套接字的内存空间中写入表示这一初始状态的控制信息。到这里，创建套接字的操作就完成了。

# 一、[[Socket]]在协议栈中的作用

 socket，其实它**不是协议**。

是：

> **操作系统提供给应用程序的网络接口抽象。**

---

## 1. 网络协议栈结构

经典 TCP/IP 结构：

```
应用层
  HTTP / DNS / FTP
      ↑
Socket API
      ↑
传输层
  TCP / UDP
      ↑
网络层
  IP
      ↑
链路层
  Ethernet / WiFi
```

socket位于：

```
应用程序  ↔  协议栈
```

---

## 2. Socket的本质

Socket 是：

> **应用层访问传输层协议的编程接口**

程序不能直接操作TCP/IP协议栈。

必须通过：

```
socket()
connect()
send()
recv()
close()
```

这些API。(操作系统中的特权指令 )

---

## 3. Socket的抽象模型

一个 socket 可以理解为：

```
( IP地址 , 端口号 , 协议 )
```

例如：

```
TCP socket
192.168.1.10 : 8080
```

完整连接是：

```
四元组

(源IP, 源端口, 目的IP, 目的端口)
```

例如：

```
客户端
10.0.0.2:53000

服务器
8.8.8.8:80
```

---

## 4. Socket在数据流中的位置

发送数据流程：

```
应用程序
  write()

socket层
  ↓

TCP层
  分段 + 序号 + ACK

IP层
  封装IP包

链路层
  MAC帧

物理网络
```

接收时反向。

---

## 5. 为什么需要socket


**屏蔽协议栈复杂性。**

应用程序只看到：

```
字节流
```

而看不到：

```
TCP重传
拥塞控制
IP路由
分片
MAC地址
```

 **内核协议栈完成**

---

## 6. TCP socket vs UDP socket

|类型|特点|
|---|---|
|TCP socket|面向连接|
|UDP socket|无连接|


TCP：

```
socket()
connect()
send()
recv()
```

UDP：

```
socket()
sendto()
recvfrom()
```

---

## 7. 更深层理解（系统视角）

Socket是：

> **操作系统中的通信端点（communication endpoint）**

每个socket在内核中对应一个 **socket结构体**：

包含：

```
协议类型
发送缓冲区
接收缓冲区
状态机
连接信息
```

TCP socket还维护：

```
序列号
ACK
窗口
拥塞状态
```

# 四、一个更本质的理解（网络哲学）

可以把互联网抽象为三层角色：

|层|角色|
|---|---|
|应用程序|使用网络|
|Socket API|操作系统接口|
|协议栈|实现通信|

socket就是：

```
应用世界 ↔ 网络世界
```

之间的 **桥梁接口**。

---

这段文字其实讲的是**Socket通信的发起阶段**，并没有涉及状态机解析（那是DNS报文或HTTP解析的内容）。我来帮你梳理这段的核心过程，同时解释其中隐含的状态流转。

---

## 一、整体过程梳理

### 宏观类比：寄存行李

```
应用程序（浏览器）
    ↓ 调用
Socket库（工具箱）
    ↓ 调用具体组件
协议栈（OS内核）
    ↓ 实际执行
网络硬件
```

浏览器自己不懂网络，它只是**按顺序调用Socket库里的组件**，把脏活累活交给下层。

---

## 二、具体步骤拆解

### 第一步：创建套接字 `socket()`

```
浏览器调用 socket()
    → 控制权转入Socket库内部
    → 协议栈在内核中分配资源，创建套接字数据结构
    → 返回一个【描述符 fd】给浏览器
    → 控制权交回浏览器
```

**描述符的本质：** 就是一个整数编号（如 fd=3），相当于"行李牌"。内核维护一张表：

```
fd=3  →  套接字A（状态：未连接，缓冲区空...）
fd=4  →  套接字B（另一个窗口的连接）
```

浏览器只需持有这个编号，后续所有操作都凭这个编号找到对应套接字。

---

## 三、套接字本身就是一个状态机

这是这段文字没有明说、但最核心的隐含结构。套接字从创建到销毁，经历一系列**明确的状态转移**：

```
        socket()调用
            ↓
    ┌───────────────┐
    │   CLOSED      │  套接字刚创建，什么都没发生
    └───────┬───────┘
            │ connect()
            ↓
    ┌───────────────┐
    │   SYN_SENT    │  发出SYN包，等待服务器响应
    └───────┬───────┘
            │ 收到SYN-ACK
            ↓
    ┌───────────────┐
    │  ESTABLISHED  │  三次握手完成，可以收发数据
    └───────┬───────┘
            │ write()/read()循环
            ↓
    ┌───────────────┐
    │  数据传输中   │  send/recv不断进行
    └───────┬───────┘
            │ close()
            ↓
    ┌───────────────┐
    │  FIN_WAIT /   │  四次挥手过程
    │  CLOSE_WAIT   │
    └───────┬───────┘
            ↓
    ┌───────────────┐
    │   TIME_WAIT   │  等待确保对端收到最后ACK
    └───────┬───────┘
            ↓
    ┌───────────────┐
    │   CLOSED      │  套接字销毁，资源释放
    └───────────────┘
```

**每一次Socket库调用，本质上就是触发这个状态机的一次转移。**

---

## 四、Socket库调用序列 与 状态机的对应

```
应用程序调用        状态机事件              套接字状态变化
─────────────────────────────────────────────────────
socket()      →  分配资源             NONE → CLOSED
connect()     →  发出SYN              CLOSED → SYN_SENT → ESTABLISHED
write()       →  发送数据             ESTABLISHED（循环）
read()        →  接收数据             ESTABLISHED（循环）
close()       →  发出FIN              ESTABLISHED → FIN_WAIT → TIME_WAIT → CLOSED
```

描述符 fd 就是这个状态机实例的"句柄"——你告诉内核 fd=3，内核就找到对应的状态机，执行对应的转移动作。

---

## 五、多套接字并发的本质

```
浏览器窗口1  →  fd=3  →  状态机实例A（连接 google.com）
浏览器窗口2  →  fd=4  →  状态机实例B（连接 github.com）
```

两个状态机**独立运行，互不干扰**。内核用描述符区分它们，就像酒店用号码牌区分不同客人的行李。

---

## 六、一句话总结

> **Socket通信 = 应用程序按顺序调用库函数，驱动内核中套接字状态机依次转移；描述符是状态机实例的编号，使多个并发连接得以独立管理。**

Socket库对应用程序隐藏了TCP状态机的所有复杂性，让浏览器只需关心"我要连谁、发什么、收什么"，其余交给协议栈。
