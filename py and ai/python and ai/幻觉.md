# 模型幻觉（Hallucination）

**定义**：模型编造不存在的事实、引用、逻辑
**本质**：概率空间中的随机游走

> [!info] 一句话本质
> 大模型是基于概率的状态机——当下一个 token 的条件概率分布过于平坦（熵高）时，输出等同于随机游走，而非"知识检索"。

## 容易产生幻觉的场景

- **长尾知识**：训练数据中出现频率极低的事实（小众论文、冷门人物、特定 API 版本）
- **时效性信息**：知识截止日期之后的事件、价格、人事变动
- **精确数字**：具体统计数据、引用页码、电话号码、代码行号
- **引用溯源**：文章作者、论文标题、链接 URL（模型会"合理捏造"）
- **专有名词组合**：两个真实概念的跨域组合（"某公司 A 的某技术 B"）
- **否定与排除**：模型在处理"不是""没有""除了"时的逻辑错误率更高
- **多步推理链**：每步引入的小误差会在链式推导中放大

## 为什么降低 Temperature 不能根治幻觉

- **Temperature 只控制采样分布的锐度**，不改变模型的参数权重与"记忆"——错误知识以高置信度被编码时，低温只会让模型更"坚定"地输出错误答案
- **Greedy decoding（T=0）同样幻觉**：argmax 只保证局部最优，模型会流畅地生成内部一致但事实错误的序列
- **根本矛盾**：幻觉的来源是训练数据的覆盖盲区与参数压缩损失，这是权重层面的问题；Temperature 是推理层面的超参，两者作用于不同层次

> [!warning] 类比
> 把一本印错了的教科书用更小的字体重印，字迹更清晰，但错误内容依然存在。

## 解决幻觉的主要方法

> [!tip] 核心思路
> 让模型多查资料、多调系统，别让模型裸答（减少模型纯依赖参数记忆的场景）

### 1. RAG — 检索增强生成

- 将用户 query 向量化，在知识库中检索相关片段，拼入 prompt 作为上下文
- 模型回答时有"原材料"可依据，减少凭空捏造
- 典型工具：LangChain、LlamaIndex、Dify；向量库：Chroma、Milvus、Faiss
- 关键点：检索质量 > 生成质量，chunk 策略和 embedding 选择至关重要

### 2. 搜索增强（Search Augmentation）

- 实时调用搜索引擎（Bing、Google、Tavily）获取最新网页内容
- 解决 RAG 静态知识库的时效问题
- 适用场景：新闻、股价、最新政策、实时赛事

### 3. 数据库查询（Text-to-SQL / Tool Use）

- 模型生成 SQL 或调用结构化查询工具，从关系型/图数据库取精确数值
- 避免模型"估算"数字——精确数字必须来自权威数据源
- 需防范 SQL 注入与权限越界，通常配合沙箱执行

### 4. API 调用（Function Calling）

- 模型将意图转化为结构化函数调用，由系统执行后将结果返回模型
- OpenAI Function Calling / Anthropic Tool Use 的核心价值之一
- 适用：天气、地图、计算器、金融行情、企业内系统

### 5. 其他补充手段

- **思维链（CoT）**：让模型显式推理，降低跳跃性错误；但不能消除事实性幻觉
- **自我一致性（Self-Consistency）**：多路采样取多数投票，提升稳定性
- **事实核验模型**：用独立模型对输出进行二次验证（如 FActScore 框架）
- **置信度引导**：在 prompt 中要求模型标注不确定性，对低置信内容追加来源
- **RLHF / 偏好对齐**：在训练阶段对幻觉行为进行惩罚（治本，但成本高）
